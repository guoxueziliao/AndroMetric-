# 0.2.7 女性性健康关怀与周期性生活规划

> 本文定义 0.2.7 如何把现有月经 / 生殖健康内容接入关系上下文系统。目标是关怀、尊重和共同规划，不是控制女性伴侣。

## 核心基调

0.2.7 可以成为男性用户理解女性伴侣性健康和性生活体验的工具。

它应帮助用户：

- 了解伴侣当前可能处在经期、预计窗口期、备孕 / 避孕目标或恢复期。
- 在性生活前更主动地沟通身体状态、边界和舒适度。
- 在性生活后记录反馈、恢复和需要照顾的事项。
- 将性生活计划和伴侣周期状态放在一起看，而不是只从男性表现出发复盘。

它不能变成：

- 对女性伴侣的监控。
- 对性行为时机的要求。
- 对月经、排卵或怀孕状态的医疗判断。
- 用周期状态推断伴侣意愿或性欲。

## 已有数据基础

当前代码已经有 reproductive / menstrual 数据：

- `ReproductiveProfile`
- `CycleEvent`
- `PregnancyEvent`
- `MenstrualDailySummary`
- period start / end。
- flow / cramp / spotting。
- ovulation test。
- fertile window estimate。
- pregnancy test / bleeding / pain / clinical visit。
- reproductive goal：仅记录、共同备孕、暂不怀孕、已怀孕、恢复期。

0.2.7 应优先复用这些数据，不重复造一套女性周期模型。

## 关怀维度

### 周期状态

允许进入上下文：

- 经期。
- 预计经期。
- 预计窗口期。
- 排卵试纸峰值。
- 痛经 / 经量 / 点滴出血。
- 恢复期。

呈现方式必须带不确定性：

- “预计窗口期。”
- “基于已有记录估算。”
- “数据不足。”
- “请以伴侣实际感受为准。”

### 性生活计划

允许支持：

- 避开伴侣不舒服的日期。
- 经期或痛经时优先提示休息、关怀和沟通。
- 备孕目标下提示共同计划和记录，但不制造压力。
- 暂不怀孕目标下提示保护措施和沟通。
- 性生活后记录伴侣反馈、恢复和是否需要后续照顾。

不允许：

- “排卵期必须安排性生活。”
- “经期不能 / 必须做什么”的绝对判断。
- “为了备孕必须增加频率。”
- “伴侣今天应该有性欲。”

### 体验关怀

关系上下文可以记录：

- 本次是否充分沟通。
- 伴侣是否舒适。
- 是否需要放慢节奏。
- 是否有疼痛、疲劳、紧张或压力。
- 事后是否需要安抚、休息、补水或后续沟通。

这些信息只用于关怀和复盘，不用于评价伴侣或判断关系。

## 复盘使用

允许输出：

- “本次记录处在预计窗口期，复盘仅作上下文提示。”
- “本次记录伴随痛经 / 不适信息，建议优先关注舒适度和恢复。”
- “当前目标是暂不怀孕，性生活记录应保留保护措施和沟通信息。”
- “伴侣反馈样本不足，只展示事实。”

禁止输出：

- “这是最佳同房时机。”
- “你应该安排性生活。”
- “伴侣处于窗口期所以更适合性生活。”
- “伴侣不适导致你的表现下降。”
- “根据记录判断伴侣怀孕 / 排卵。”

## UI 入口候选

- Reproductive / 月经面板中的“性生活计划提醒”。
- 性行为记录中的周期上下文提示。
- 成人行为复盘中的女性性健康上下文片段。
- 伴侣详情中的生殖健康 / 周期关怀区。
- 关系沟通目标中的周期关怀 check-in。

所有入口都应允许关闭或跳过。

## Safety 边界

必须显示或隐含这些原则：

- 周期估算不是医疗判断。
- 女性伴侣的实际感受优先于任何记录。
- 性生活计划必须以沟通和同意为前提。
- 备孕 / 避孕目标不能被系统转化成压力。
- 私密周期记录默认不进入 CSV / 报告类导出。

## 非目标

- 不做女性健康诊断。
- 不做怀孕结论。
- 不做排卵确定性判断。
- 不做性生活排程强提醒。
- 不做催促、打卡或任务化性生活。
- 不把周期状态与伴侣价值绑定。
