# 0.2.6 范围与边界

> 本文负责 0.2.6 的核心基调、版本目标和范围边界。

## 核心基调

0.2.6 的核心基调为：

> 让复杂本地数据长期可信、可检查、可恢复。

0.2.2 - 0.2.5 连续增加成人行为事件、复盘事实、训练目标和目标历史。0.2.6 应回答：

- 新增数据是否都进入完整 JSON backup。
- 导入旧包或新包时，用户是否能看懂风险。
- 数据关联坏掉时，完整性检查是否能发现。
- 用户是否能在不真的覆盖当前数据前做恢复预检。

0.2.6 不应该把产品推向账号系统、云同步或多设备协作。

## 版本目标

0.2.6 可以做：

- JSON backup 完整契约审计。
- import preview 扩展。
- snapshot integrity 扩展。
- linked ids / orphan / one-way relation 检查。
- 训练目标和 check-in 完整性检查，仅在真实 store / 类型已落地后纳入。
- 只读恢复预检。
- 备份写后读信心反馈。
- CSV 可读导出边界整理，并移除 Markdown 导出。

## 默认数据边界

0.2.6 默认不新增 schema。

优先复用：

- 现有 `SnapshotData` / `ExportSnapshot`。
- 现有 import preview。
- 现有 snapshot integrity。
- 0.2.2 三类成人行为事件。
- 0.2.4 `TrainingGoal` / `GoalCheckin`，仅在真实 store / 类型已落地后复用。
- 0.2.5 目标历史的运行时派生结果。

如果实现时发现必须新增 schema，应停止本版本规划并重新讨论。

## 明确不做

- 后端同步。
- 账号系统。
- 云端备份。
- 多设备实时合并。
- CRDT / OT。
- 外部 AI 分析。
- 自动内容识别。
- 自动修复业务事实。
- 自动删除用户历史。
- 分享图或社交传播。

## 和 0.1.1 / 0.1.3 的区别

0.1.1 / 0.1.3：

- 建立 JSON / CSV / Markdown 等导出入口；其中 Markdown 后续由 0.2.6 决策取消。
- 建立导入预览。
- 建立自动备份、quota、保留策略和安全快照。

0.2.6：

- 针对 0.2.2 - 0.2.5 新增复杂数据做完整性补强。
- 重点不是增加备份入口，而是补齐复杂数据的契约、预检和检查。
- Markdown 导出不再延续，当前版本应移除或隐藏入口。
- 不重复做自动备份设置、quota 设置或保留策略 UI。
