# 0.2.8 一致性审计 2026-05-29

> 本文记录 0.2.8 从研究草案收口为执行草案前的文档一致性审计。

## 状态

- 版本状态：执行草案 / 待实现。
- 主方向：个人常态系统 v1。
- 实现前置：必须先完成刀 1 真实代码校准。
- 默认不新增 schema、store、cache 或导出字段。

## 已一致

- 所有文档都以用户自己的历史为基线。
- 默认 baseline window 为 90 天。
- current window 为 14 / 30 天。
- 用户可见 confidence 最高为 medium。
- 输出状态只保留三类。
- 派生结果只服务 UI。
- 0.2.5 / 0.2.7 相关指标只在真实落地后读取。

## 刀序一致性

- 刀 1：代码校准。
- 刀 2：基线派生模型。
- 刀 3：个人常态工作台。
- 刀 4：偏离提示与记录缺口。
- 刀 5：Safety / Privacy 审计。

实现窗口不得跳过刀 1。

## 安全边界

一致不做：

- 医学诊断。
- 异常检测。
- 人群标准比较。
- 未来预测。
- 伴侣归因。
- 总分 / 等级 / 排名。
- 分享图或对外报告。

## 数据边界

- `null` 表示不可计算或缺失。
- 可信 `0` 只用于次数 / 负荷类指标的无事件日。
- scalar 指标缺失不得填 0。
- JSON backup 仍只负责原始数据恢复。
- CSV 默认不包含个人常态派生结果。

## 待实现窗口确认

实现时仍需确认：

- `StatsEngine` 和 `eventAdapter` 的当前行为。
- 0.2.3 review facts 的真实可复用范围。
- 0.2.5 目标历史是否已落地。
- 0.2.7 关系上下文是否已落地。
- UI 入口应挂在 Stats 哪个现有区域下。

这些属于代码校准，不需要重新扩展产品范围。

## 审计结论

0.2.8 文档已经可作为执行草案交给实现窗口。若真实代码与文档不一致，以真实代码为准缩小范围，并同步回改本组文档。
