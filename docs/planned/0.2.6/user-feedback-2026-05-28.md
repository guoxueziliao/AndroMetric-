# 0.2.6 用户反馈插刀（2026-05-28）

> 本文记录 0.2.6 规划期间发现的现有数据生态可用性问题。它只描述需求和验收，不声明代码已实现。

## 反馈 1：导出需要明确“全部导出”

现状：

- 导出选项展示开始日期和结束日期。
- 用户需要的是全部导出，而不是被日期区间暗示为必须筛选。

需求：

- 导出默认应为全部导出。
- 日期筛选应是可选高级筛选，而不是默认心智。
- UI 需要明确展示“全部导出”状态。
- 用户清空日期后，应回到全量导出。

验收：

- 打开导出弹窗时，默认是全部导出。
- 用户能一眼知道当前导出范围是全部数据。
- 只有用户主动选择日期筛选时，开始 / 结束日期才影响导出。
- JSON backup 的完整迁移导出不应被误导成日期区间导出。

## 反馈 2：数据生态提示“一键修复”但找不到按钮

现状：

- 数据生态区会提示“建议运行一键修复”。
- 用户在数据生态区找不到对应按钮。
- 实际修复按钮位于数据健康区，且依赖体检结果和 `canRepair`。

需求：

- 数据生态提示必须能直接带用户到可操作位置。
- 如果一键修复按钮当前不可用，应说明原因。
- 如果必须先运行数据健康检查，应在提示旁提供“去体检”或“开始体检”动作。
- 修复入口不应只藏在另一个区块。

验收：

- 数据生态区出现修复建议时，用户能直接触发或跳转到修复流程。
- 未运行体检时，提示“先运行数据健康检查”。
- `canRepair=false` 时，不显示“建议一键修复”这类不可执行提示。
- 一键修复按钮和提示在同一个用户路径里闭合。

## 反馈 3：数据健康问题跳转后不知道对应什么

现状：

- 数据健康检查能列出问题。
- 点击问题只跳转到对应日期表单。
- 用户不知道具体是哪个字段、哪条自慰/性爱/素材记录或哪个子项有问题。

需求：

- 问题列表必须展示具体定位信息。
- 跳转到表单时，应带上字段路径或可理解的目标。
- 如果无法自动聚焦，也要展示“需要检查的位置”。

定位信息至少包括：

- 日期。
- 问题类型。
- 严重程度。
- 字段路径或业务位置。
- 对应记录索引或子项索引。
- 建议动作。

验收：

- 健康问题列表不只显示日期和泛化消息。
- 对 `masturbation[0].contentItems[1]` 这类问题，UI 能展示“自慰记录 1 / 素材 2”或等价文案。
- 点击问题后，用户知道应该检查表单里的哪个区块。
- 无法自动定位的 issue，也要显示清楚的人工检查路径。

## 归属

这三项并入 0.2.6：

- 全部导出：归刀 5 CSV / Markdown 可读导出边界，同时影响 JSON export UX。
- 修复入口可发现：归刀 3 Snapshot Integrity 扩展和刀 6 Safety / Privacy 审计。
- 健康问题定位：归刀 3 Snapshot Integrity 扩展，必要时作为 UI 小插刀。
