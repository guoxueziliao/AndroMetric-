# 0.2.6 插刀 0 验收矩阵

> 本文只记录插刀 0 的实现顺序和验收场景。它不替代 0A / 0B / 0C 细节文档。

## 实现顺序

建议顺序：

1. 0A 导出范围 UX。
2. 0B 修复入口状态流。
3. 0C 健康问题定位。

原因：

- 0A 最独立，风险最低。
- 0B 只整理现有体检 / 修复入口。
- 0C 可能需要调整 issue path 文案和跳转提示，放最后更稳。

## 0A 导出范围

必须验证：

| 场景 | 预期 |
|---|---|
| 打开 JSON 导出 | 默认全部导出 |
| 打开加密 JSON 导出 | 默认全部导出 |
| 打开 CSV 导出 | 默认全部导出 |
| Markdown 导出入口 | 已移除或隐藏 |
| 切到按日期并选择范围 | 只导出日期范围内数据 |
| 切回全部 | 日期不再影响导出 |
| 不填写日期 | 不阻止导出 |

失败信号：

- 用户仍被要求填写日期。
- UI 看不出当前是全部还是按日期。
- JSON 完整备份被日期筛选误导。

## 0B 修复入口

必须验证：

| 状态 | 数据生态应展示 |
|---|---|
| 未体检 | 开始体检 |
| 已体检且可修 | 运行一键修复 |
| 已体检但不可修 | 查看问题详情 |
| 已体检且无问题 | 数据健康，无需修复 |

失败信号：

- 出现“建议一键修复”但没有按钮。
- `canRepair=false` 仍提示一键修复。
- 修复绕过确认或安全快照。

## 0C 健康问题定位

必须验证：

| path | 预期文案 |
|---|---|
| `masturbation[0].contentItems[1]` | 自慰记录 1 / 素材 2 |
| `masturbation[0].contentItems[1].platform` | 自慰记录 1 / 素材 2 / 来源平台 |
| `sex[0].partner` | 性爱记录 1 / 伴侣 |
| `exercise[0]` | 运动记录 1 |
| `sleep.startTime` | 睡眠 / 开始时间 |
| unknown path | 展示原始 path 和人工检查提示 |

失败信号：

- issue 只显示日期。
- issue 只显示泛化消息。
- 点击后不知道检查哪个区块。

## 最低验证命令

实现后至少跑：

```bash
npm run test
npm run typecheck
git diff --check
```

如果改动导出弹窗、数据健康列表或设置弹层，应补一次浏览器手测。
