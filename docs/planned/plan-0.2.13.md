## 当前状态

- 状态：执行草案 / 待实现。
- 前置：0.2.12 已确定。
- 当前实现线：0.2.6 正在开发。
- 0.2.5 及以前的完成版本不进入本轮规划维护。
- 主方向：把写死在每日记录里的补剂，升级为可管理、可暂停、可回看的健康项目周期系统。
- 第一批对象：补剂；模型不只为补剂设计，应能扩展到晒太阳、拉伸、康复、作息干预等非医疗健康项目。
- 已定边界：补剂进入第一版；有周期计划的健康习惯可以进入第一版；用药只保留为每日健康事实或备注，不进入 0.2.13 周期计划系统。

## 讨论原则

- 先确定健康项目、补剂、用药和习惯的边界，再决定是否进入执行草案。
- 不继承旧 0.2.13 档案索引方向。
- 不反向扩大 0.2.5 - 0.2.12 已定范围。
- 若新 0.2.13 依赖 0.2.6 - 0.2.12 的真实实现结果，必须在文档中显式标注。

## 初步范围

- 健康项目档案：名称、类型、目标、备注和状态。
- 周期计划：每日、隔日、每周若干天、连续 N 天、开始日期、可选结束日期、暂停 / 恢复和手动结束。
- 执行记录：每日应做项、完成、跳过、备注。
- 补剂字段：剂量、单位、服用时间和开始 / 结束 / 暂停。
- 补剂详情：名称、可选剂量、单位、服用时间标签、周期和备注。
- 历史范围：项目详情做事实回看；Dashboard 只提示今日；阶段回顾后续只读概况。
- 每日记录：不再维护写死补剂列表，只显示当前计划生成的待记录项。
- 用药记录：保留为每日健康记录事实，不做计划、提醒、剂量管理或医嘱判断。
- 入口位置：每日记录健康区作为主入口；Dashboard 只做轻提示；不放入“我的 / 设置”，不新增顶级导航。

## 明确不做

- 药物医嘱系统。
- 疾病治疗计划。
- 自动推荐补剂。
- 剂量安全判断。
- 备孕成功率、性表现或健康结果的强因果判断。
- 根据补剂类型自动推荐周期。
- 成就、连续打卡、排名、评分或补剂效果分析。
- 医学诊断、用药建议或替代医生意见。

## 已定依赖与兼容

- 新 0.2.13 大概率需要 schema / migration；具体表结构实现前按代码校准。
- 新 0.2.13 作为 0.2.12 后的独立版本；如果 0.2.6 后置反馈池先处理，则以实现窗口安排为准。
- 新 0.2.13 不强依赖 0.2.7 - 0.2.12 真实落地，但必须等待 0.2.6 数据安全能力稳定后再实现。
- 第一版健康项目类型固定为 `supplement`、`sunlight`、`stretching`、`rehab`、`sleep_routine`、`other_habit`。
- 当前 daily log 的 `supplements` 字段作为 legacy daily fact 继续兼容读取，不自动迁移成长期计划。

## 推荐实现顺序

1. 代码校准：确认 `DailyLog.supplements`、导入导出、数据健康和生殖面板读取点。
2. 数据模型：新增健康项目、计划和执行记录；配套 migration。
3. 项目管理：创建、编辑、暂停、结束、归档。
4. 每日记录接入：用计划生成今日待记录项，旧补剂记录兼容展示。
5. 数据安全接入：JSON backup、import preview、snapshot integrity。
6. Safety / Privacy 审计：清除医疗建议、剂量安全判断和服药提醒风险。

## 暂不做

- 不恢复旧 0.2.13 档案索引方向。
- 不在本窗口改产品代码。
