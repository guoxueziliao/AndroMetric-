# 0.2.4 研究入口（养成系与关系/表现训练）

> 本文档用于 0.2.4 的讨论研究。
> 当前主方向暂定：养成系正式化 + 关系与性表现训练。
> 0.2.3 做轻量养成 v0 和成人行为复盘；0.2.4 在此基础上把“复盘之后怎么改进”做成长期体验。

## 当前状态

- 状态：研究入口。
- 前置：0.2.2 成人行为事件模型；0.2.3 洞察与复盘增强。
- 暂定主方向：养成系正式化、关系与伴侣维度深化、性表现训练、恢复训练。
- 已定：养成范围先做“训练建议 + 进度反馈 + 轻目标”，不做完整游戏化等级系统。
- 已定：允许新增最小数据模型，但必须先完成训练数据模型文档，再开发 schema / migration。
- 已定：性表现训练首批覆盖硬度稳定、做爱时长、射精控制观察、满意度、事后恢复，但展示优先级不同。
- 已定：关系与伴侣维度进入 0.2.4 主线，但只做“关系上下文 + 沟通/反馈 + 表现影响观察”，不做完整关系管理系统。
- 已定：做轻量规则型建议系统，不叫 AI，不做复杂引擎；只要 AI 不能本地部署，AI 功能就不做。
- 已定：轻目标采用“系统推荐，用户手动确认”，不自动创建目标。
- 已定：建立 Safety Rails，从目标类型、建议规则、文案、指标展示四层避免养成系统诱导高风险性行为。
- 已定：开发刀序和验收标准。
- 已定：模型细节：生理日规则、标题可编辑、本周期感受、7 / 14 天周期、单来源 insight。
- 已定：入口位置：成人行为复盘作为主入口，Dashboard 做轻提示，不单独做 Training 页面。
- 已定：`description` / `note` / `cycleFeeling` 的 UI 暴露方式：快速流程不展示长文本，“本周期感受”非必填。
- 待定：0.2.1 收尾后是否有 UI 组件接口变化需要同步到 0.2.4。

## 核心基调

0.2.4 的核心基调暂定为：

> 从复盘走向养成：让性健康、做爱表现、恢复和关系质量形成可持续改进路径。

0.2.2 让数据可记录，0.2.3 让数据可复盘，0.2.4 应该回答：

- 用户看完复盘之后下一步做什么？
- 如何改善硬度稳定性、做爱表现、恢复状态和关系体验？
- 如何让记录习惯、恢复习惯、沟通习惯和性表现训练长期持续？

0.2.4 不应做成刺激用户追求更多性次数、更多伴侣或更多射精的游戏系统。养成系必须服务健康、表现、关系和恢复。

## 已定养成范围

0.2.4 选择：

> 先做“训练建议 + 进度反馈 + 轻目标”，不做完整游戏化等级系统。

原因：

- 0.2.3 已经有轻量养成 v0，0.2.4 应在此基础上让用户知道“下一步怎么改进”。
- 完整等级、徽章、任务系统会扩大范围，也容易诱导用户追求次数和刺激，而不是健康与恢复。
- 性健康和关系表现不是普通游戏目标，反馈需要克制、可解释、可退出。

### 做什么

0.2.4 可以做：

- 训练建议：根据复盘结果给出可执行的下一步建议。
- 进度反馈：展示硬度稳定性、恢复质量、记录质量、做爱表现稳定性、关系沟通等趋势。
- 轻目标：允许用户选择少量健康导向目标。
- 目标回顾：每周 / 每月查看目标是否改善。
- 风险提示：连续疲劳、低恢复、低硬度、高负荷时提示暂停或恢复。

### 不做什么

0.2.4 不做：

- 完整等级系统。
- 成就徽章体系。
- 每日任务列表。
- 连续挑战。
- 排行榜。
- 伴侣数量目标。
- 性次数目标。
- 射精次数目标。
- 色情使用时长目标。
- 鼓励高风险性行为的任何目标。

### 目标类型

允许的目标类型：

- 记录质量：补全硬度、睡眠、疲劳、满意度等关键字段。
- 恢复：改善睡眠、降低疲劳、减少高负荷后连续透支。
- 表现稳定性：观察硬度波动、做爱时长波动、满意度波动。
- 射精控制观察：记录边缘控制、射精时间、事后疲劳和恢复，不做次数挑战。
- 关系沟通：记录边界、偏好、事后反馈和沟通备注。

禁止的目标类型：

- 多约炮。
- 多伴侣。
- 多射精。
- 更长色情使用。
- 更高色情刺激。
- 伴侣比较或排名。
- 忽视同意、安全、恢复的目标。

### 表达规则

允许：

- “近 14 天恢复波动较大，建议优先稳定睡眠记录。”
- “做爱表现数据不足，继续记录 3 次性行为后再看趋势。”
- “高负荷后疲劳偏高，本周目标可设为记录恢复和睡眠。”

不允许：

- “升级成更强男人。”
- “挑战一周更多次性爱。”
- “提高射精次数。”
- “刷更多色情内容探索偏好。”
- “某伴侣让你表现变差。”

## 已定数据模型边界

0.2.4 选择：

> 允许新增最小数据模型，但必须先写模型文档，再开发 schema / migration。

执行前置文档：

- [training-data-model-0.2.4.md](./training-data-model-0.2.4.md)

### 为什么允许新增

0.2.4 的核心不是继续展示更多复盘结论，而是让用户从复盘走向长期改进。如果完全不新增模型，训练建议、进度反馈和轻目标只能临时依附在现有记录上，很难形成稳定闭环。

但 0.2.4 也不能直接做完整养成系统。完整的等级、计划、任务、徽章、连续挑战会扩大范围，并且容易把产品引向性次数、射精次数、色情刺激或伴侣数量的追逐。

因此本版本只允许支持 0.2.4 必需闭环的最小模型。

### 允许的最小概念

0.2.4 优先允许：

- `TrainingGoal`：用户当前关注的轻目标，例如硬度稳定、恢复、记录质量、关系沟通、射精控制观察。
- `GoalCheckin`：用户对轻目标的周期回顾，例如本周是否改善、是否继续、是否暂停。

0.2.4 暂不优先持久化：

- `TrainingSuggestion`：建议优先由规则即时生成，除非后续讨论确认需要保存历史建议。
- `TrainingPlan`：暂不做完整训练计划，除非后续确认需要连续周期管理。
- `Milestone` / `Badge` / `Level`：不做等级和徽章。

### schema 规则

如果后续确认新增表或字段，必须满足：

- 先完善 `training-data-model-0.2.4.md`。
- 再修改 `core/storage/db.ts`。
- 同步修改 `core/storage/migration.ts`。
- 导入 / 导出 / 快照需要能保留新数据。
- 不破坏既有历史数据。
- 不新增云端、账号、同步、审核或分享相关字段。

### 禁止的数据建模

0.2.4 不建模：

- 伴侣排名。
- 伴侣分数。
- 性次数挑战。
- 射精次数挑战。
- 色情使用时长挑战。
- 色情刺激等级追逐。
- 成瘾判定。
- 医学诊断。
- 社交传播或分享。

## 已定性表现训练维度

0.2.4 选择：

> 首批覆盖硬度稳定、做爱时长、射精控制观察、满意度、事后恢复，但展示优先级不同。

这意味着 0.2.4 不是只做单一硬度训练，也不是直接做完整性技巧课程。首批目标是把做爱表现中最核心、最可记录、最能和既有健康数据联动的维度接起来。

### 展示优先级

#### 1. 硬度稳定

硬度稳定是 0.2.4 性表现训练的第一优先级。

关注：

- 做爱过程中硬度是否稳定。
- 是否出现中途变软、需要暂停、需要更强刺激才能维持。
- 硬度波动与睡眠、疲劳、酒精、压力、色情使用、自慰、射精频率之间的关系。
- 与 0.2.3 的硬度 / 勃起质量复盘打通。

表达方向：

- “近期硬度稳定性较低，优先观察睡眠、疲劳和色情使用后的表现变化。”
- “样本不足，继续记录 3 次性行为后再判断硬度趋势。”

不表达：

- “硬度越高越好。”
- “挑战更强硬度。”
- “某个伴侣导致你表现差。”

#### 2. 事后恢复

事后恢复是第二优先级，用来防止训练系统诱导透支。

关注：

- 性行为后的疲劳。
- 次日晨勃。
- 次日精神状态。
- 睡眠质量。
- 连续性行为 / 自慰 / 色情使用后的恢复压力。
- 高负荷后是否需要暂停或降低强度。

表达方向：

- “高负荷后恢复偏慢，本周建议优先观察睡眠和疲劳。”
- “连续记录显示事后疲劳较高，建议暂停追求表现数据，先看恢复。”

不表达：

- “继续挑战更多次。”
- “恢复越快说明能力越强。”
- “低恢复就是失败。”

#### 3. 射精控制观察

射精控制进入首批，但定位为观察，不做挑战。

关注：

- 射精时间。
- 是否过快或过久。
- 主观控制感。
- 边缘控制是否使用。
- 边缘控制后的疲劳、硬度、满意度和恢复。
- 与自慰、色情使用、压力、睡眠的关系。

表达方向：

- “射精控制波动较大，建议记录控制感、疲劳和恢复，而不是单看时长。”
- “边缘控制后疲劳偏高，建议观察恢复窗口。”

不表达：

- “挑战更久。”
- “坚持越久越好。”
- “提高射精次数。”
- “训练成某种固定表现标准。”

#### 4. 做爱时长

做爱时长进入首批，但不作为越长越好的目标。

关注：

- 时长稳定性。
- 是否明显短于自身常态。
- 是否明显长到影响疲劳或满意度。
- 时长与硬度、射精控制、满意度、恢复的匹配。

表达方向：

- “本阶段更适合观察时长稳定性，而不是追求更长。”
- “时长变化需要和满意度、硬度、恢复一起看。”

不表达：

- “越久越好。”
- “挑战最长做爱时间。”
- “把时长作为核心能力排名。”

#### 5. 满意度

满意度进入首批，作为表现质量和关系体验的补充指标。

关注：

- 自己的主观满意度。
- 伴侣反馈。
- 关系氛围。
- 是否有沟通、边界、偏好和事后反馈。
- 满意度与硬度、时长、射精控制、恢复之间的关系。

表达方向：

- “满意度需要和关系氛围、沟通、恢复一起看。”
- “本次表现数据正常，但满意度较低，建议记录沟通和偏好因素。”

不表达：

- 伴侣评分。
- 伴侣排名。
- “谁让你表现更好 / 更差。”
- 对关系形态做道德判断。

### 首批不做

0.2.4 首批不做：

- 完整性技巧库。
- 系统训练课程。
- 固定周期训练营。
- 以时长、次数、射精数量为目标的挑战。
- 面向伴侣的评分系统。
- 医学治疗建议。

原因：

- 当前版本更需要打通记录、建议、轻目标和回顾闭环。
- 技巧库和训练课程内容量大，容易偏离本地健康管理主线。
- 性表现训练必须优先避免诱导透支、比较和风险行为。

## 已定关系与伴侣维度

0.2.4 选择：

> 关系与伴侣维度进入 0.2.4 主线，但只做“关系上下文 + 沟通/反馈 + 表现影响观察”，不做完整关系管理系统。

这个决定的核心是：性表现不是单纯身体数据。硬度、做爱时长、射精控制、满意度和事后恢复都会受到关系氛围、沟通、边界、偏好、熟悉度、压力和伴侣反馈影响。0.2.4 如果完全不处理关系上下文，训练建议会缺少重要背景。

但 0.2.4 也不能扩张成完整关系管理产品。本版本只把关系维度接入性健康、性表现和养成闭环。

### 进入主线的内容

0.2.4 可以做：

- 性行为记录关联关系 / 伴侣上下文。
- 多伴侣、多性别伴侣、多关系形态的中性表达。
- 性表现复盘中展示关系上下文。
- 满意度拆成自我满意度、伴侣反馈、沟通质量等轻量字段。
- 训练目标支持“关系沟通”类目标。
- 建议规则可以提示沟通、边界、偏好和事后反馈，但不能下判断。

### 表现影响观察

0.2.4 可以观察：

- 不同关系上下文下的硬度稳定性。
- 沟通质量与满意度之间的关系。
- 伴侣反馈与自我满意度是否一致。
- 压力、紧张、关系氛围与射精控制、硬度波动之间的关系。
- 事后反馈与恢复状态之间的关系。

表达方式必须是观察式：

- “在沟通记录更完整的性行为中，满意度样本更稳定。”
- “近期关系氛围备注较少，满意度变化暂时无法解释。”
- “伴侣反馈和自我满意度不一致时，建议补充沟通记录。”

不允许表达成强因果或评判：

- “某个伴侣导致你表现变差。”
- “某类伴侣更适合你。”
- “这个关系不健康。”
- “你应该更换伴侣。”

### 多伴侣与多性别伴侣表达

0.2.4 必须避免默认异性恋、单偶、一男一女关系。

文案和字段命名应支持：

- 单一伴侣。
- 多伴侣。
- 多性别伴侣。
- 固定关系。
- 非固定关系。
- 开放关系 / 多偶关系。
- 未标记关系。

推荐表达：

- “伴侣 / 关系上下文”
- “本次关系氛围”
- “伴侣反馈”
- “沟通与边界”

避免表达：

- “女友 / 老婆 / 女性伴侣”作为默认。
- “正经关系 / 不正经关系”。
- “主伴侣更好 / 临时伴侣更差”。
- 对关系形态做道德评价。

### 0.2.4 不做完整关系管理系统

0.2.4 不做：

- 完整伴侣档案系统重做。
- 关系阶段管理。
- 关系协议管理。
- 关系事件时间线。
- 冲突 / 修复记录系统。
- 伴侣偏好资料库。
- 伴侣评分。
- 伴侣排名。
- 伴侣优劣判断。
- 自动判断“适不适合某伴侣”。

这些内容可以作为 0.2.5+ 的候选方向，但不进入 0.2.4 的开发范围。

### 与轻目标的关系

关系维度可以进入轻目标，但只能是健康、沟通和复盘导向。

允许目标：

- “本周补充 2 次事后反馈。”
- “记录关系氛围和沟通质量。”
- “观察满意度和沟通记录是否一致。”
- “记录边界、偏好或不适点。”

不允许目标：

- “尝试更多伴侣。”
- “比较不同伴侣表现。”
- “找出让你表现最好的伴侣。”
- “提高征服感。”
- “用伴侣反馈排名关系质量。”

## 已定建议系统边界

0.2.4 选择：

> 做轻量规则型建议系统，不叫 AI，不做复杂引擎。只要 AI 不能本地部署，AI 功能就不做。

### 核心原则

0.2.4 的建议系统只做本地规则判断。它不是 AI 助手，不是诊断系统，也不是复杂 recommendation engine。

原因：

- 当前产品是隐私优先、本地优先 PWA。
- 成人行为、性健康、色情使用、伴侣关系数据高度敏感。
- 如果 AI 需要云端推理，就会破坏本地私密产品的核心边界。
- 规则型建议已经足够支撑 0.2.4 的训练建议、轻目标和进度反馈。

### AI 边界

0.2.4 不做：

- 云端 AI。
- 调用外部大模型 API。
- 把成人行为、色情使用、性表现、伴侣关系数据发出本地。
- “AI 分析你的性生活”之类的功能命名。
- AI 聊天、AI 教练、AI 诊断、AI 伴侣建议。

长期规则：

> 只要 AI 不能本地部署，AI 功能就不做。

如果未来要重新讨论 AI，必须满足：

- 完全本地运行。
- 不上传隐私数据。
- 用户可关闭。
- 输出可解释。
- 不做医学诊断。
- 不做伴侣判断。
- 不做成瘾判定。
- 不做诱导性行为或色情消费的建议。

### 0.2.4 建议系统形态

0.2.4 的建议采用即时生成，默认不持久化。

每条建议应包含：

- `dimension`：适用维度，例如硬度、恢复、射精控制、时长、满意度、关系沟通、记录质量。
- `trigger`：触发原因，例如样本不足、疲劳偏高、恢复偏慢、沟通记录缺失。
- `confidence`：可信度或样本量提示。
- `message`：用户可读建议。
- `nextAction`：可执行下一步。

建议不需要保存为数据库实体，除非后续确认必须追踪建议历史。

### 首批规则方向

允许的规则：

- 样本不足时，建议继续记录，不下结论。
- 硬度稳定性低且疲劳高时，优先建议恢复和睡眠记录。
- 做爱时长波动大但满意度稳定时，提示不必只盯时长。
- 满意度低且沟通记录缺失时，建议补充关系氛围和伴侣反馈。
- 色情使用后 24-48 小时硬度波动明显时，建议继续观察窗口，不直接判定因果。
- 高负荷后恢复偏慢时，建议暂停追求表现数据，优先记录恢复。
- 射精控制波动大时，建议同时看控制感、疲劳和恢复，而不是单看时长。

禁止的规则：

- 诊断 ED、早泄、成瘾或心理问题。
- 判断某个伴侣导致表现好或差。
- 建议增加性次数、射精次数、伴侣数量或色情使用时长。
- 给出强因果结论。
- 用羞辱、焦虑、竞争、征服感驱动用户行为。
- 生成成人挑逗文案。

### 命名规则

推荐命名：

- “建议”
- “训练建议”
- “下一步”
- “本周关注”
- “观察提示”

避免命名：

- “AI 建议”
- “智能教练”
- “AI 性生活分析”
- “诊断结果”
- “能力评分”

## 已定轻目标创建方式

0.2.4 选择：

> 系统推荐，用户手动确认；不自动创建目标。

规则型建议系统可以根据复盘结果生成轻目标候选，但不能替用户创建目标。涉及性表现、色情使用、自慰、伴侣关系和恢复状态时，系统如果自动创建“你该改什么”的目标，会让产品显得冒犯，也会削弱本地私密工具的控制感。

### 基本流程

复盘完成后：

1. 系统最多展示 1-3 个候选轻目标。
2. 用户可以选择“开始”“忽略”“稍后”。
3. 只有用户点击“开始”后，才创建 `TrainingGoal`。
4. 用户可以编辑目标标题、周期和备注。
5. 目标到期后进入 `GoalCheckin`。
6. 用户在 check-in 中选择继续、暂停、完成或调整。

### 默认推荐优先级

推荐轻目标时，默认优先级为：

1. 恢复。
2. 硬度稳定。
3. 记录质量。
4. 射精控制观察。
5. 关系沟通。
6. 做爱时长。

原因：

- 恢复优先可以避免养成系统诱导透支。
- 硬度稳定是男性健康主轴。
- 记录质量决定后续复盘可信度。
- 射精控制、关系沟通和做爱时长更依赖样本和上下文，不适合在数据不足时强推。

### 高风险 / 低恢复规则

如果近期出现以下情况：

- 连续疲劳偏高。
- 事后恢复偏慢。
- 睡眠明显不足。
- 硬度明显下降且样本不足。
- 色情使用、自慰、性行为负荷连续偏高。

系统只能推荐：

- 恢复类目标。
- 记录质量类目标。
- 暂停追求表现数据的观察提示。

系统不能推荐：

- 表现提升目标。
- 做爱时长目标。
- 射精控制挑战。
- 更多性行为或更多色情使用相关目标。

### 目标周期

0.2.4 默认目标周期建议：

- 7 天：适合记录质量、恢复观察、短期高负荷后恢复。
- 14 天：适合硬度稳定、射精控制观察、关系沟通。
- 30 天：暂不作为默认周期，只作为后续增强候选。

周期不应被设计成连续打卡挑战。到期后只做回顾，不做失败惩罚。

### 用户控制

用户必须可以：

- 忽略候选目标。
- 稍后再看。
- 修改标题。
- 修改周期。
- 添加备注。
- 暂停目标。
- 归档目标。

系统不允许：

- 自动创建目标。
- 自动开启训练。
- 自动判定失败。
- 自动推送羞辱或焦虑式提醒。
- 因用户忽略目标而降低评分。

### 推荐文案

允许：

- “近期恢复波动较大，可以开始一个 7 天恢复观察目标。”
- “硬度稳定性样本不足，可以先做 14 天记录质量目标。”
- “伴侣反馈记录较少，可以选择一个关系沟通观察目标。”

不允许：

- “系统已为你创建硬度提升目标。”
- “你应该马上训练射精控制。”
- “不开始目标会影响你的能力评分。”
- “挑战更久做爱时间。”
- “本周必须完成更多性行为记录。”

## 已定 Safety Rails

0.2.4 选择：

> 建立 Safety Rails：目标类型、建议规则、文案、指标展示四层限制。

Safety Rails 是 0.2.4 的开发验收条件，不是普通文案建议。任何目标、建议、卡片、指标、空状态、toast、confirm 或复盘文案，只要诱导更多性行为、更多射精、更高色情刺激、伴侣比较或忽视恢复，都不能进入版本。

### 第一层：目标类型限制

允许目标：

- 记录质量。
- 恢复。
- 硬度稳定。
- 性表现稳定性观察。
- 射精控制观察。
- 关系沟通。
- 伴侣反馈记录。

禁止目标：

- 性次数目标。
- 射精次数目标。
- 伴侣数量目标。
- 色情使用时长目标。
- 色情刺激强度目标。
- 做爱最长时间挑战。
- 更多约炮 / 更多伴侣目标。
- 忽视同意、安全、恢复的目标。

高负荷 / 低恢复时：

- 只能推荐恢复类目标。
- 只能推荐记录质量类目标。
- 可以提示暂停追求表现数据。
- 不能推荐表现提升、做爱时长、射精控制挑战或更多性行为相关目标。

### 第二层：建议规则限制

建议规则必须满足：

- 恢复优先。
- 样本不足时只提示继续记录，不下结论。
- 所有建议必须带触发原因。
- 所有趋势判断必须考虑样本量和可信度。
- 做爱时长、射精控制、性次数必须和满意度、硬度、恢复一起看。
- 色情使用相关建议只能表达为观察窗口，不能直接判定因果。
- 伴侣 / 关系相关建议只能表达为沟通、边界、反馈和上下文补充。

建议规则禁止：

- 医学诊断。
- 成瘾判定。
- 强因果结论。
- 自动判断某个伴侣导致表现好或差。
- 建议增加性次数、射精次数、伴侣数量或色情使用时长。
- 用羞辱、焦虑、竞争、征服感驱动用户。

### 第三层：文案限制

允许文案风格：

- 直白成人健康。
- 自我管理。
- 观察式。
- 可执行。
- 不道德化。
- 不羞辱。

允许例子：

- “近期恢复偏慢，建议优先记录睡眠和疲劳。”
- “样本不足，继续记录后再看趋势。”
- “射精控制需要和疲劳、满意度、恢复一起观察。”
- “伴侣反馈缺失时，满意度变化暂时不建议下结论。”

禁止文案风格：

- 挑逗化。
- 羞辱。
- 竞争。
- 征服感。
- 能力排名。
- 贩卖焦虑。
- 伪医学诊断。

禁止例子：

- “升级成更强男人。”
- “挑战更多次性爱。”
- “击败上一次的做爱时长。”
- “找出最适合你的伴侣。”
- “色情刺激越强越能训练能力。”
- “恢复慢说明能力差。”

### 第四层：指标展示限制

指标展示必须满足：

- 硬度和恢复优先于次数、时长、射精数量。
- 做爱时长不做最高纪录。
- 射精控制不做挑战榜。
- 性次数不做连续打卡。
- 色情使用时长不做累计成就。
- 伴侣维度不做排名、评分或胜负比较。
- 表现指标必须和满意度、硬度、恢复一起展示。
- 低恢复时，表现指标降级展示，恢复提示优先。

禁止展示：

- “最长做爱纪录。”
- “本月性次数挑战。”
- “射精次数统计成就。”
- “色情使用时长排行。”
- “最佳伴侣 / 最差伴侣。”
- “让你表现最好的伴侣。”

### 验收条件

0.2.4 每个相关刀必须检查：

- 是否引入了被禁止的目标类型。
- 是否存在诱导更多性行为、更多射精或更高色情刺激的建议。
- 是否把伴侣做成排名、评分或优劣判断。
- 是否在低恢复状态下继续强调表现提升。
- 是否出现羞辱、竞争、征服、能力排名或焦虑文案。
- 是否用强因果、诊断或成瘾判定解释数据。

只要任一项命中，不能合入 0.2.4。

- 最近一个 active 目标。
- 即将到期的 check-in。
- 低恢复 / 高负荷时的恢复提示。
- 一条简短的“本周关注”。

Dashboard 不展示：

- 完整目标列表。
- 完整建议列表。
- 训练课程。
- 任务面板。
- 连续打卡。
- 性次数 / 射精次数 / 色情使用时长挑战。

### 不做独立 Training 页面

0.2.4 不新增独立 Training 页面。

不做原因：

- 当前版本重点是建立闭环，不是建立完整训练中心。
- 独立页面会诱导做更多目标、更多计划、更多管理状态。
- 训练建议来源于复盘，主入口离开复盘会削弱上下文。

后续只有在以下条件满足时，才重新讨论独立 Training 页面：

- 用户同时存在多个 active / paused / archived 目标。
- 需要查看目标历史。
- 需要跨周期比较 check-in。
- 需要训练计划或长期周期。
- Dashboard 轻提示和复盘入口承载不住。

## 已定轻目标与 Check-in UI 细节

0.2.4 选择：

> `TrainingGoal.description` 不在快速创建里显示，只在完整编辑里显示；`GoalCheckin.note` 不在快速 check-in 里显示，只在完整编辑里显示；`cycleFeeling` 非必填。

### TrainingGoal.description

0.2.4 支持 `TrainingGoal.description`，但快速创建不显示。

规则：

- 快速创建只显示默认标题、目标类型、周期和触发原因。
- 用户点击完整编辑时，才展示 description。
- description 用于用户补充目标背景，不参与规则建议。
- description 不能用于生成被 Safety Rails 禁止的目标。

原因：

- 手机 Chrome 是高频场景，快速创建必须短。
- 目标说明是补充信息，不应该挡住“开始 / 忽略 / 稍后”的主决策。

### GoalCheckin.note

0.2.4 支持 `GoalCheckin.note`，但快速 check-in 不显示。

规则：

- 快速 check-in 只显示状态选择：继续、暂停、完成、调整。
- 完整编辑才显示 note。
- note 用于记录本周期背景，不参与自动诊断。
- note 不用于生成伴侣评分、能力评分或强因果结论。

原因：

- 到期 check-in 应该是低摩擦动作。
- 备注适合事后补充，不适合作为必填项。

### cycleFeeling

`cycleFeeling` 表示 1-5 的“本周期感受”，但不是必填。

规则：

- 用户可以只选择状态，不填写本周期感受。
- 如果填写，展示文案使用“本周期感受”。
- 不叫评分，不叫表现评分，不叫能力评分。
- 不用于排名、惩罚或失败判断。

原因：

- 主观感受有价值，但不应把 check-in 变成考试。
- 用户在疲劳、低恢复、关系压力高时，仍应能快速完成回顾。
