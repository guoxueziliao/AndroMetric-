## 核心决策

- 0.2.3 的核心基调：让成人行为、色情使用和健康状态形成可解释的复盘系统。
- 0.2.3 解决“记录之后能看见什么”，不解决医学诊断、成瘾判定或强因果结论。
- `adult behavior review engine` 只做时间关系、趋势、共现、事实摘要、样本量提示和弱相关观察。
- review engine 不直接读 Dexie，不依赖 React，不把 UI 文案写死在模型内部。
- 复盘首屏围绕硬度、勃起质量、恢复、睡眠和行为负荷。
- 色情使用、自淫 / 自慰、性行为、射精、边缘控制等数据是解释健康状态的上下文，不把产品做成单纯色情使用统计器。
- 周报 / 月报是本地复盘报告，不是分享内容；永远不做分享图。
- Markdown 导出允许成人、露骨、直白词，但导出前必须提醒包含敏感成人健康数据。
- 所有洞察必须包含 `sampleSize` 和 `confidence`；样本不足时只展示事实，不输出判断。
- 0.2.3 默认最多输出到 `medium` confidence，避免复盘系统过度自信。
- 实现顺序：先事实时间线和窗口聚合，再做弱相关观察，最后做报告和导出。
- 入口形态：嵌入现有统计 / 洞察区域，不新增顶级导航。
- 数据边界：0.2.3 不新增 schema / migration，使用 0.2.2 已完成的数据闭环。
- 实现交接：实现窗口先按 adapter → facts → confidence → insights → UI / report 的数据流校准，不在 engine 内直接读 Dexie。
- Review input：adapter 只整理 Porn use / Masturbation / Sex / daily log 等纯数据，不生成判断。
- 事实时间线：按 `targetDate` 分组、按 `startedAt` / 稳定 `sortKey` 排序，linked ids 只展示链路，不推断因果。
- 首屏信息架构：优先显示窗口、硬度 / 恢复、行为负荷、缺失数据、insight preview 和 timeline preview。
- 弱相关观察：第一批只覆盖睡眠与晨间硬度、性活动负荷与疲劳、Porn use 与次日硬度、Masturbation、Sex、记录质量。
- Markdown 报告：使用本地事实数据和通过 confidence gating 的 insight，不默认包含 notes 全文、具体平台名或任何内容本体。
- 刀 50：先做代码入口校准，不实现业务逻辑，确认 StatsEngine / UI / export / storage 入口后再进入刀 51。

## 明确不做

- 医学诊断。
- ED / 成瘾 / 性功能障碍判定。
- “X 导致 Y”的强因果结论。
- 道德化评价。
- 羞辱性或训诫式语言。
- 云端模型或联网分析。
- 复杂黑箱模型。
- 样本不足时包装成洞察。
- 分享图。
