# 0.2.7 范围与边界

> 本文负责 0.2.7 的核心基调、版本目标和范围边界。

## 核心基调

0.2.7 的核心基调为：

> 记录关系情境，关怀女性伴侣的性健康和周期状态；不评价伴侣，不诊断关系。

0.2.4 让关系上下文进入训练建议，0.2.5 回看轻目标历史，0.2.6 收紧数据安全。0.2.7 正式把这条线命名为关系上下文与女性性健康关怀系统 v1，应回答：

- 一次性行为发生在什么关系情境里？
- 哪些沟通、边界、反馈和恢复背景值得记录？
- 女性伴侣当前的月经、窗口期、备孕 / 避孕目标或恢复状态如何影响性生活计划？
- 伴侣资料里哪些长期信息真的能服务复盘和沟通？
- 旧的伴侣姓名引用和新的 stable id 如何共存？
- 如何记录边界、偏好和反馈，而不变成评分、比较或关系管理平台？

## 可以做

- 关系上下文 v1 模型。
- 女性性健康关怀与周期性生活规划。
- 单次记录中的关系情境采集。
- 伴侣资料信息架构整理，作为长期上下文来源。
- 伴侣身份引用契约审计。
- legacy sex record partner name 与 stable partner id 的兼容边界。
- 关系上下文摘要：关系情境、沟通反馈、边界偏好、健康 / 恢复背景。
- 成人行为复盘中的中性关系上下文提示。
- 训练目标历史里的关系沟通类回看。
- 伴侣资料隐私与导出边界整理。

## 默认不做

- 完整关系管理系统。
- 多伴侣关系图谱。
- 伴侣评分、排名、能力比较或优劣判断。
- 情感咨询、关系诊断或冲突裁决。
- 女性健康诊断、怀孕结论、排卵确定性判断。
- 用周期状态催促或安排性生活。
- 面向伴侣的分享报告。
- 自动识别或合并疑似同一伴侣。
- 云端同步、账号系统或外部 AI。

## 数据边界

0.2.7 默认优先复用现有数据：

- `PartnerProfile`
- legacy `SexRecordDetails.partner`
- legacy `SexInteraction.partner`
- adult behavior `SexEvent.partnerIds`
- reproductive `partnerId`
- `CycleEvent`
- `PregnancyEvent`
- `MenstrualDailySummary`

如果实现中需要新增字段，应优先作为 optional 字段进入现有 `PartnerProfile`，并配套导入导出、snapshot integrity 和隐私审计。

如果实现中需要自动迁移 legacy name 到 stable id，应停下来重谈。系统不能凭名字静默合并伴侣，也不能在历史记录中制造错误关联。

## 版本边界

0.2.7 是关系上下文与女性性健康关怀系统 v1，不是关系平台版本，也不是女性健康诊断版本。

完成后的状态应该是：

- 单次性行为记录可以表达关系情境。
- 月经 / 生殖健康数据能作为性生活关怀和计划上下文。
- 伴侣资料更适合长期维护，但不是使用关系上下文的前置条件。
- 关系上下文能被复盘和训练历史安全读取。
- 用户能理解哪些信息会被用于上下文提示。
- 没有伴侣评分、排名或“谁更好”的表达。
