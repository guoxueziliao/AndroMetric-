# 0.2.21 修复安全与备份

## 复用现有安全模式

`features/profile/model/useProfileMaintenance.ts` 的 `handleRepairData` 已建立可复用的安全流程：

1. 用户确认。
2. `StorageService.snapshots.create(..., 'auto-safety')` 建修复前快照。
3. 执行修复 / 清理。
4. 写 `db.meta` 标记（如 `dataFixVersion`）。
5. Toast 反馈处理条数。
6. 重新 `runHealthCheck`。

本版的全表修复与孤立清理必须沿用这套流程，不另起一套不带快照的清理路径。

## 安全约束

- **修复 / 清理前必建快照**：任何会改写或删除数据的动作，前置自动快照不可省略。
- **可撤销**：用户能从修复前快照恢复；快照机制见 `core/storage/snapshotIntegrity.ts`。
- **不静默**：每类清理需明确确认，不默认执行、不一键全删。
- **结果可见**：处理条数与重新体检结果必须反馈给用户。
- **失败可中断**：异常时给出错误反馈并停止，不留下半修复状态（沿用现有 try/finally）。

## 与 0.2.6 数据安全的关系

0.2.6 已建立备份 / 快照 / 长期数据承诺。0.2.21 是其消费方：

- 不重做备份机制，直接复用快照能力作为清理前置。
- 不改备份 / 导出的 JSON 字段名（遵守 CLAUDE.md：不为重构改用户可见字段）。

## 验收

- 全表修复与孤立清理前都创建了 `auto-safety` 快照。
- 可从该快照恢复到清理前状态。
- 无任何绕过快照的清理路径。
- 清理失败时不留半状态。
