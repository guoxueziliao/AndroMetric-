
# UI 状态语义规范 (Final)

**状态**: ✅ Final / 可长期使用
**适用版本**: v0.0.6 起（向后兼容）

## 一、设计目标 (Why)

本系统是长期自用的数据记录系统，不是展示型产品。
UI 中的所有状态词，必须满足以下目标：

1.  **半年后再看，仍然一眼懂**
2.  不依赖颜色、图标或记忆
3.  不使用抽象、玩梗或技术性词汇
4.  所有状态都能明确回答一个事实问题

---

## 二、核心判断模型 (三分法，定死)

所有 UI 状态，必须明确区分以下三类之一：

| 状态词 | 含义 | 是否发生 | 是否有数据 | 是否可判断 |
| :--- | :--- | :---: | :---: | :---: |
| **已发生 / 有记录** | 行为发生且已记录 | ✅ | ✅ | ✅ |
| **未发生** | 明确没有发生 | ❌ | ✅ | ✅ |
| **未记录** | 可能发生，但没有记录 | ❓ | ❌ | ❌ |
| **无** | 当前维度不适用 | ❌ | ❌ | ✅ |

⚠️ **禁止新增第四种否定语义**

---

## 三、四种状态词的强制使用规则

### 1️⃣ 已发生 / 有记录

*   **使用条件**: 行为确实发生，或数据已填写
*   **示例**:
    *   自慰：已发生
    *   性爱：已发生
    *   夜间睡眠：6 小时

### 2️⃣ 未发生 (❗行为专用)

*   **使用条件**: 用户**明确**没有做，系统可确认行为未发生
*   **允许使用的模块**: 自慰, 性爱, 运动, 白天睡眠（午睡 / 打盹）
*   **示例**:
    *   今日自慰：未发生
    *   白天睡眠：未发生
*   **❌ 禁止用于**: 睡眠总览, 数据完整性提示

### 3️⃣ 未记录 (❗数据专用)

*   **使用条件**: 理论上应该有数据，但用户没有填写
*   **允许使用的模块**: 夜间睡眠, 健康状态, 心理状态
*   **示例**:
    *   夜间睡眠：未记录
    *   心理状态：未记录
*   **❌ 禁止替代为**: 0, 无, 一般, Avg / Active

### 4️⃣ 无 (❗上下文不适用)

*   **使用条件**: 当前维度不在讨论范围，或统计为空
*   **示例**:
    *   本月性伴侣：无
    *   今日异常：无
*   **📌 “无” ≠ 没发生**

## 四、睡眠模块的最终结构定义 (定死)

**睡眠 = 夜间睡眠 + 白天睡眠**（两个独立语义）

### 夜间睡眠 (状态 + 数据)
| 情况 | 显示 |
| :--- | :--- |
| 有记录 | 夜间睡眠：6 小时 |
| 未填写 | 夜间睡眠：**未记录** |

**❌ 禁止**: `0h`, `Avg`, `Quality`

### 白天睡眠 (行为)
| 情况 | 显示 |
| :--- | :--- |
| 有记录 | 白天睡眠：2 次 · 共 40 分钟 |
| 明确没睡 | 白天睡眠：未发生 |
| 不确定 | 白天睡眠：**未记录** |

---

## 五、主界面「今日状态」卡片规范

*   **用途**: 快速确认今天做了什么，不是数据完整性检查
*   **允许状态**: 已发生, 未发生
*   **示例**:
    ```
    今日状态
    ────────────
    运动：已完成
    自慰：已发生
    性爱：未发生
    ```
*   **❌ 禁止**: 未记录, Active / Done / Avg

---

## 六、全系统禁用词表 (永久)

| 禁用词 | 原因 | 替代 |
| :--- | :--- | :--- |
| **Active** | 抽象，不可回忆 | 已发生 |
| **Done** | 不说明内容 | 已完成 |
| **Avg** | 技术统计词 | 一般 / 不显示 |
| **Quality** | 模糊 | 睡眠情况 / 评分 |

---

## 七、强制原则 (给未来的你)

UI 文案必须服务于 **事实回忆**，
而不是风格、情绪或设计感。
